# Code review главной страницы — фокус на стабильность

Дата: 2026-06-07
Скоуп: вся главная (`apps/web/src/pages/index.astro` и все блоки/острова/рантаймы, на которые она опирается)
Приоритет ревью: стабильность — «чтобы работало надёжно и не падало»
Метод: многоагентное ревью по 6 измерениям (рантайм-жизненный цикл, React-острова,
контракт Loader↔Hero, адаптив/кроссбраузерность, сборка/зависимости, доступность).
52 сырых находки → каждую перечитывал отдельный агент-скептик (адверсариальная
проверка) → **32 подтверждено, 20 отсеяно** как ложные. Серьёзность ниже
выставлена с поправкой на реальный, а не теоретический, риск.

## Итог

- **Сборка здоровая:** `astro check` — 0 ошибок / 0 предупреждений / 0 подсказок (19 файлов);
  `astro build` проходит, static-вывод, ~1.8s.
- **Рантайм-контракты в основном крепкие.** Адверсариальный проход отклонил большинство
  «утечек», которых нет: Lenis сам чистит слушатели в `destroy()`; `cleanup()` лоудера
  null-безопасен (снимает `loading` до обращения к `wrap`); React-эффекты корректно
  отписываются на unmount; `getBoundingClientRect().left` не «плывёт» при вертикальном
  скролле; `agentation` mount идемпотентен. Это значит — проблемы точечные, не системные.
- **2 критичных пункта живут в общей обвязке** (`BaseLayout` + связка Loader↔Hero),
  поэтому утекут на каждую следующую страницу. Их разумно закрыть до перехода к новым страницам.

## 🔴 Критично

### 1. Гейт лоудера отключён (TEST-маркер) — лоудер играет на каждой загрузке
- **Файл:** `apps/web/src/layouts/BaseLayout.astro:36-39`
- **Проблема:** условие `!sessionStorage.getItem("taiji:loaded") &&` закомментировано
  (TEST-маркер). Класс `html.loading` ставится на **каждой** загрузке, а не один раз
  за сессию.
- **Влияние:** при любом заходе/перезагрузке проигрывается ~5с лоудер + блокировка
  скролла (`html.loading { overflow:hidden }`). Это `BaseLayout` — бьёт по всем будущим
  страницам. Дополнительно усиливает баг №2.
- **Фикс:** раскомментировать строку с проверкой `sessionStorage`.

### 2. Hero может навсегда остаться пустым — гонка раскрытия
- **Файлы:** `apps/web/src/components/HeroSlider.tsx:50-59`,
  `apps/web/src/lib/loader.ts:99,112`
- **Проблема:** лоудер диспатчит `taiji:loader-done` **рано** — в
  `HOLD_UNTIL + EXIT - REVEAL_LEAD ≈ 4.2с`, а класс `loading` снимает только в
  `finish()` на `≈5.0с`. Образуется окно **~0.8с**, где событие уже улетело, но класс
  ещё висит. Если `HeroSlider` (`client:load`) гидрируется именно в этом окне
  (медленное устройство/сеть), то на маунте видит `loading` → вешает **одноразовый**
  слушатель `taiji:loader-done` → событие уже не повторится (`revealSignaled` блокирует)
  → `revealed` остаётся `false` навсегда.
- **Влияние:** текст hero (`[data-rise]`, скрыт через `html.js [data-rise]{opacity:0}`)
  и фон-видео (`.play()` под `if(!revealed) return`) **не появляются вообще**.
  Фолбэка нет. Баг №1 увеличивает экспозицию (лоудер теперь на каждой загрузке).
- **Фикс:** сделать сигнал «статичным» — в `loader.ts` при диспатче выставлять
  стойкий маркер (напр. `document.documentElement.classList.add("loader-done")` или
  `window.__taijiLoaderDone = true`), а в `HeroSlider` на маунте проверять
  `if (нет loading || маркер выставлен) setRevealed(true); else addEventListener`.
  Плюс страховочный таймаут (см. №3).

## 🟠 High

### 3. Нет фолбэка раскрытия / риск вечного FOUC
- **Файлы:** `apps/web/src/components/HeroSlider.tsx:50-59`, `apps/web/src/components/Hero.astro:142`
- **Проблема:** раскрытие hero завязано на единственную доставку события
  `taiji:loader-done`. Если оно не пришло по любой причине (ошибка в таймлайне, race
  из №2), `[data-rise]` остаются `opacity:0` навсегда.
- **Фикс:** в `HeroSlider` добавить страховочный таймер — принудительный
  `setRevealed(true)` через ~6с, если reveal так и не наступил.

### 4. `init()` анимационного рантайма без обработки ошибок
- **Файл:** `apps/web/src/lib/smooth.ts:120-135` (и `startSmoothScroll`/`setup*`-функции)
- **Проблема:** `init()` последовательно вызывает `startSmoothScroll()`, `setupReveals()`,
  `setupBgReveals()`, `setupStaggeredReveals()`, `setupHeroParallax()`, `ScrollTrigger.refresh()`
  без `try/catch`. Один throw обрывает весь рантайм для этой загрузки и оставляет
  частично инициализированное состояние.
- **Влияние:** GSAP/Lenis падают редко, но в духе «не падало» — дёшево застраховаться.
  При сбое лучше деградировать (нет smooth-скролла), чем «убить» все reveal-анимации.
- **Фикс:** обернуть тело `init()` (или каждую setup-функцию по отдельности) в `try/catch`
  с логом; при ошибке вызывать `destroy()` для чистки частичного состояния.

### 5. Мобильная вёрстка ломается *(в скоупе — по решению; см. примечание)*
- **Файлы:** `apps/web/src/components/HeroSlider.tsx:140` (h1 `text-[60px] whitespace-nowrap`
  обрезается на узких экранах), `apps/web/src/components/AboutInfo.tsx:64`
  (`w-[377px]` переполняет < 440px), а также отсутствие брейкпоинтов по всему hero
  (`text-[28px]`, `w-[260px]` и т.д.).
- **Влияние:** на телефонах hero-заголовок обрезается (страница не скроллится вбок
  из-за `overflow-hidden`, но текст режется); плашка AboutInfo выходит за вьюпорт.
- **Примечание:** страница сейчас десктоп-fixed целиком. Это вопрос скоупа — есть
  отдельные мобильные артборды в design.pen (390px, гаттеры 15px). Чинить точечно
  или закладывать полный адаптив — решение пользователя.

## Что проверено и оказалось НЕ проблемой (ложные срабатывания)

Зафиксировано, чтобы не возвращаться: эти контракты сделаны корректно.

- Lenis: `lenis.on("scroll", ...)` без сохранения отписки — **ок**, `destroy()` чистит
  всех слушателей (`emitter.destroy()`).
- `cleanup()` лоудера (`loader.ts:115-131`) — **null-безопасен**: снимает `loading`
  до обращения к `wrap`; залипания scroll-lock нет.
- `setupHeroParallax()` — ранний `return` при отсутствии `[data-hero-darken]` — корректная защита.
- React-эффекты (`AboutInfo`, `HeroSlider`) — cleanup гарантированно вызывается на unmount
  под `<ClientRouter />`; «осиротевших» слушателей `document` нет.
- `getBoundingClientRect().left` в `ResearchSlider` — не «плывёт» при вертикальном скролле.
- `requestIdleCallback(run, {timeout:500})` — таймаут это потолок ожидания, а не задержка;
  лоудер всегда стартует.
- `key={index}` в слайдерах — штатно перезапускает CSS-анимацию на каждой смене слайда.
- Имена классов (`loading`/`js`/`hero-revealed`) синхронны во всех 5 точках использования.

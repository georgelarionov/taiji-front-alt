# Параллакс hero — дизайн / handoff

Дата: 2026-06-01
Статус: дизайн одобрен, параметры зафиксированы — ожидает реализации в следующей сессии

## Обзор

Эффект «контент наезжает на hero». Hero-блок (`1-block`) становится неподвижным
нижним слоем; при скролле второй блок и всё, что ниже, едут поверх него (выше по
z-index), попутно hero **затемняется**. Hero остаётся на месте (не двигается) —
это не «медленный» параллакс-сдвиг, а перекрытие неподвижного фона.

Зафиксированные параметры (из уточнения у пользователя):

| Параметр | Решение |
|---|---|
| Степень затемнения к концу | **Среднее, ~50%** (hero остаётся различим сквозь затемнение) |
| Что затемняется | **Весь hero целиком** — видео, текст, меню, контролы под единым слоем |
| Дистанция затемнения | **За первый экран** (~100svh скролла, к моменту полного перекрытия) |

## Механика перекрытия (чистый CSS, без JS)

- Корневой `<section>` острова `HeroSlider` (`HeroSlider.tsx:77`) меняем
  `relative` → `sticky top-0 z-0`. Высота остаётся `h-svh min-h-[640px]`.
  `position: sticky; top: 0` прилепляет hero к верху вьюпорта; он держит свой
  слот в потоке (100svh) и остаётся неподвижным фоном на всё время скролла.
- Весь контент после `<Hero />` в `index.astro` оборачиваем в
  `<div class="relative z-10 bg-white"> … </div>` (`About`…`Footer`). Это слой
  выше hero по z-index → он наезжает поверх неподвижного hero. У блоков уже свои
  непрозрачные фоны; `bg-white` на обёртке — подстраховка от просвета hero в щель.

**Почему sticky, а не `position: fixed` или ScrollTrigger pin:** sticky сам
резервирует место в потоке (не нужен ручной спейсер под fixed) и не добавляет
pin-spacer'ы ScrollTrigger'а, которые капризничают в связке с Lenis. Lenis
использует нативный скролл (не transform) → `sticky` работает без конфликтов.

## Затемнение (GSAP scrub в `smooth.ts`)

Внутрь hero-секции добавляем слой затемнения — **поверх всего** содержимого
hero (фон-видео `-z-20`, скрим `-z-10`, текст/меню/контролы — дефолтный z):

```tsx
{/* Слой затемнения параллакса: едет 0 → 0.5 за первый экран (GSAP в smooth.ts).
    Поверх всего hero (z-30) → темнеет весь блок целиком. */}
<div
  data-hero-darken
  aria-hidden="true"
  className="pointer-events-none absolute inset-0 z-30 bg-ink opacity-0"
/>
```

`bg-ink` = токен `--color-ink` (#0d1014, не чистый чёрный). Старт `opacity-0`.

Новая функция в `smooth.ts`, регистрируется в `init()` (после `startSmoothScroll()`,
перед `ScrollTrigger.refresh()`):

```ts
function setupHeroParallax() {
  const overlay = document.querySelector<HTMLElement>("[data-hero-darken]");
  const hero = overlay?.closest("section");
  if (!overlay || !hero) return;
  gsap.fromTo(
    overlay,
    { opacity: 0 },
    {
      opacity: 0.5,
      ease: "none",
      scrollTrigger: {
        trigger: hero,
        start: "top top",   // скролл 0
        end: "bottom top",  // низ hero у верха вьюпорта ≈ 100svh
        scrub: true,
      },
    },
  );
}
```

`scrub: true` привязывает opacity к прогрессу скролла линейно; за первый экран
(start→end = 0→100svh) слой едет 0 → 0.5, ровно к моменту полного перекрытия.

**Важно про sticky + ScrollTrigger:** start/end считаются от позиции триггера в
потоке (слот 100svh у верха документа), а не от «прилипшей» позиции — sticky
резервирует слот, поэтому маппинг скролла 0→100svh корректен. Pin самим
ScrollTrigger'ом НЕ используем (перекрытие делает CSS sticky).

## Lifecycle (контракт проекта)

Всё живёт в общем рантайме `smooth.ts`: слушатели на верхнем уровне модуля,
`astro:page-load` → `init()` (там вызвать `setupHeroParallax()`),
`astro:before-swap` → `destroy()` (уже убивает все ScrollTrigger'ы через
`ScrollTrigger.getAll().forEach(t => t.kill())` — отдельная очистка не нужна).

## Граничные случаи

- **`prefers-reduced-motion`:** `init()` в `smooth.ts` выходит раньше (строка ~73)
  → scrub не запускается, слой остаётся `opacity-0` (без затемнения). Sticky —
  чистый CSS, остаётся, но это раскладка, а не анимация → корректное поведение.
- **SPA-навигация (`ClientRouter`):** при повторном `astro:page-load` старые
  триггеры убиты в `destroy()`, `setupHeroParallax()` пересоздаёт — без «отвала».
- **Lenis:** нативный скролл → sticky и ScrollTrigger работают штатно.
- **Горизонтальный оверфлоу:** sticky не добавляет; `overflow-hidden` на секции
  уже есть. Проверить `scrollWidth ≤ innerWidth` при верификации.
- **z-index / stacking:** hero `z-0` создаёт контекст наложения; обёртка контента
  `z-10` рисуется выше. Внутренний `relative z-40` у хедера `About` локален в
  обёртке — на иерархию hero/контент не влияет.

## Верификация

- `pnpm check` — обязательно (см. префикс nvm 22 в CLAUDE.md).
- `pnpm build` — обязательно (статическая сборка проходит).
- Визуальная проверка в браузере (Playwright) — только если пользователь попросит явно.
- Ручные пункты: при скролле контент наезжает на неподвижный hero; hero темнеет
  до ~50% к концу первого экрана; на SPA-переходе и повторных загрузках эффект не
  «отваливается»; при reduced-motion затемнения нет, hero просто фон; нет
  горизонтального скролла.

## Вне scope / открытые заметки

- **Hero-видео продолжает играть под перекрытием** (минимальный CPU). В scope не
  берём. Если нужно — паузить активное видео в `HeroSlider`, когда hero полностью
  перекрыт (потребует ScrollTrigger-колбэка `onLeave`/`onEnterBack` или
  пересечения; добавит координацию внутри острова).
- Поведение на mobile/таче (sticky + smooth) отдельно не проверялось — best-effort.
- Контракт с лоудером (`taiji:loader-done`, `html.loading`) не затрагивается:
  параллакс независим от интро hero.
